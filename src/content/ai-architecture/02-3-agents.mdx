## <TermTooltip term="react">ReAct</TermTooltip>-malli (Reasoning + Acting)

ReAct on arkkitehtuuri, jossa <TermTooltip term="llm">LLM</TermTooltip> vuorottelee **päättelyn** (Reasoning) ja
**toiminnan** (Acting) välillä. Sen sijaan, että malli tuottaisi yhden
vastauksen, se käy läpi silmukan:

<AgentWorkflowDiagram />

1. **Päättely (Thought):** "Mitä minun pitää tehdä seuraavaksi?"
2. **<TermTooltip term="function-calling">Toiminta (Action)</TermTooltip>:** "Kutsun sää-API:a saadakseni tämän päivän säätiedot"
3. **Havainto (Observation):** "Sää-API palautti: Aurinkoista, 22°C"
4. → Takaisin vaiheeseen 1, kunnes vastaus on valmis

```typescript
interface AgentState {
  thought: string;
  action: ToolCall | null;
  observation: string;
  done: boolean;
}

async function reactLoop(task: string, tools: Tool[]) {
  const state: AgentState = { thought: "", action: null, observation: "", done: false };
  const messages = [
    { role: "system", content: "Olet avulias agentti. Käytä työkaluja harkiten." },
    { role: "user", content: task },
  ];

  while (!state.done) {
    const response = await llm(messages); // Päättely + toiminta
    const action = parseAction(response);

    if (action.type === "FINISH") {
      state.done = true;
      return response;
    }

    // Suorita työkalu
    state.observation = await executeTool(action);
    messages.push({ role: "assistant", content: response });
    messages.push({ role: "tool", content: state.observation });
  }
}
```

<Callout type="info">
  **ReAct vs. Chain-of-Thought:** CoT on pelkkää päättelyä (ajattele ääneen),
  ReAct on päättelyä + toimintaa. CoT sopii matematiikkaan ja logiikkaan,
  ReAct tilanteisiin, joissa tarvitaan ulkoisia työkaluja.
</Callout>

### Työkalujen käyttö

Agentin tärkein ominaisuus on kyky käyttää työkaluja (function calling).

| Työkalu | Käyttötapaus | Esimerkki |
|---|---|---|
| **Laskin** | Matemaattiset laskut | `calc("3 * 7 + 2")` |
| **Sää-API** | Reaaliaikainen säätieto | `getWeather("Helsinki")` |
| **Vektorihaku** | Tiedonhaku omasta datasta | `search(query: "verotus 2026")` |
| **Koodin suoritus** | Ohjelmointitehtävät | `runPython("print('hello')")` |
| **SQL-kysely** | Tietokantahaut | `queryDB("SELECT * FROM users")` |
| **Verkkohaku** | Reaaliaikainen internet-haku | `webSearch("uutiset tänään")` |

<AnalogyBox>
  **Analogia:** Agentti on kuin ravintolan keittiömestari. Hän:
  1. **Suunnittelee** (päättely) — "Tarvitsen alkupalan, pääruoan ja jälkiruoan"
  2. **Delegoi** (työkalut) — "Kokkaaja 1, tee alkupala. Kokkaaja 2, hoida pääruoka"
  3. **Tarkkailee** (havainto) — "Alkupala on valmis, pääruoka on uunissa"
  4. **Oppii** (muisti) — "Viimeksi asiakas piti enemmän kalasta kuin lihasta"

  Ilman agenttia (pelkkä LLM) kokki yrittäisi tehdä kaiken itse — ja
  polttaisi ruoan. Agentin kanssa kokki hallinnoi tiimiä ja varmistaa
  että kaikki valmistuu ajallaan.
</AnalogyBox>

### Monen agentin järjestelmät

Yhden agentin sijaan voidaan käyttää useita erikoistuneita agentteja.

#### Hierarkkiset järjestelmät

```
Pääagentti (orkestraattori)
├── Tutkimusagentti (hakee tietoa)
├── Koodausagentti (kirjoittaa koodia)
├── Testausagentti (testaa koodin)
└── Dokumentaatioagentti (kirjoittaa selitykset)
```

Pääagentti jakaa tehtävät, valvoo edistymistä ja yhdistää tulokset.

#### Vertaisverkot (Peer-to-peer)

Agentit kommunikoivat keskenään ilman keskusohjausta:

```
Agentti A: "Tarvitsen dataa asiakkaista"
Agentti B: "Minulla on pääsy asiakasdataan, haen sen"
Agentti C: "Minä visualisoin datan, kun saat sen"
```

#### Kommunikaatioprotokollat

| Menetelmä | Kuvaus | Esimerkki |
|---|---|---|
| **Viestipohjainen** | Agentit lähettävät toisilleen JSON-viestejä | RabbitMQ, Kafka |
| **Yhteinen tila** | Kaikki kirjoittavat jaettuun muistiin | Redis, tietokanta |
| **LLM-välitteinen** | Agentit keskustelevat LLM:n kautta | "Agentti A sanoo: ..." |

### Kriittiset ongelmat

| Ongelma | Oire | Ratkaisu |
|---|---|---|
| **Päättelysilmukka** | Agentti jumittuu samaan toimintoon | Maksimi iteraatio, timeout |
| **Hallusinaatio työkalun tuloksessa** | Agentti keksii API-vastauksen | Pakota validointi: jos API palautti 500, älä generoi "Onnistui!" |
| **Token-kustannukset** | Agentti tekee 20 turhaa API-kutsua | Rajaa päättelyä, käytä pieniä malleja osatehtäviin |
| **Kontekstin ylivuoto** | Keskusteluhistoria vie liikaa tilaa | Muistin yhteenveto, unohda vanha |
| **Työkalujen standardointi** | Jokainen agenttikehys toimii eri tavalla | Käytä OpenAI function calling -formaattia (de facto standardi 2026) |

<Callout type="warning">
  **⚠️ Kustannusansa:** Yksi 3-iteraation agenttisilmukka GPT-5:lla maksaa ~0.15-0.30 €.
  Tuhat kyselyä päivässä = 150–300 € / päivä. **Ratkaisu:**
  1. Käytä halvempaa mallia (GPT-4o-mini, Llama 4 8B) kaikkiin iteraatioihin
  2. Isompaa mallia vain viimeiseen generointivaiheeseen
  3. Harkitse: riittäisikö pelkkä huolellinen prompt engineering?
</Callout>

```typescript
// Päättelysilmukan esto + kustannusseuranta
const MAX_ITERATIONS = 10;
const MAX_COST_EUROS = 0.50;  // Maksimi per keskustelu
let totalCost = 0;

for (let i = 0; i < MAX_ITERATIONS; i++) {
  if (totalCost > MAX_COST_EUROS) {
    return fallbackResponse("Kustannusraja saavutettu");
  }
  const response = await agent.step();
  totalCost += response.cost;
  if (response.isFinal) break;
}

### Muistin hallinta

Agenteilla on kolme muistitasoa:

| Muistityyppi | Kuvaus | Esimerkki |
|---|---|---|
| **Työmuisti** (working) | Nykyisen tehtävän konteksti | Keskusteluhistoria, väliaikaiset tulokset |
| **Episodinen muisti** (episodic) | Aiemmat istunnot | "Viimeksi tämä käyttäjä kysyi X:stä" |
| **Semanttinen muisti** (semantic) | Faktoja ja tietoa | "Yrityksen nimi on ABC, perustettu 2020" |

### Kehykset (Frameworkit)

| Kehys | Kieli | Vahvuus | Heikkous |
|---|---|---|---|
| **LangChain** | Python/TS | Suuri ekosysteemi, paljon integraatioita | Monimutkainen, paljon abstraktioita |
| **LangGraph** | Python/TS | Graafipohjainen agenttien hallinta | Uusi, nopeasti muuttuva |
| **CrewAI** | Python | Yksinkertainen monen agentin malli | Rajoitettu joustavuus |
| **AutoGen** | Python | Microsoftin agenttikehys | Monimutkainen konfiguraatio |
| **Semantic Kernel** | C#/Python | Microsoftin AI-orkestraatio | Windows-painotteinen |

<WhyItMatters>
  **Miksi tällä on väliä?** Agentit ovat seuraava askel LLM-sovellusten
  kehityksessä — ne eivät vain *vastaa*, ne *toimivat*. Mutta agenttien
  rakentaminen on monimutkaista: päättelysilmukat, token-kustannukset ja
  hallusinaatiot ovat todellisia ongelmia, jotka vaativat huolellista
  arkkitehtuurisuunnittelua. Älä rakenna agenttia, jos pelkkä RAG riittää.
</WhyItMatters>
