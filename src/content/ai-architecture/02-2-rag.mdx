## Perus-RAG

Retrieval-Augmented Generation (RAG) on arkkitehtuuri, joka yhdistää
**tiedonhaun** (retrieval) ja **tekstin generoinnin** (generation).
Sen idea on yksinkertainen: sen sijaan, että kysytään LLM:ltä suoraan,
haetaan ensin relevantti konteksti omasta tietokannasta ja annetaan se
mallille mukaan.

<RAGDiagram />

RAG-putken vaiheet:

1. **Datan valmistelu:** dokumentit pilkotaan paloiksi (chunks)
2. **Upotukset (embeddings):** jokaisesta palasta luodaan vektoriesitys
3. **Indeksointi:** vektorit tallennetaan vektoritietokantaan
4. **Kysely:** käyttäjän kysymys muutetaan vektoriksi
5. **Haku:** lähimmät vektorit haetaan (kNN-haku)
6. **Kontekstin muodostus:** löydetyt palat lisätään promptiin
7. **Generointi:** LLM tuottaa vastauksen kontekstin perusteella

<Callout type="tip">
  **Avainajatus:** RAG on tapa antaa LLM:lle *pääsy omaan tietoon* ilman
  kalliista hienosäädöstä. Kun data muuttuu, päivität vektoritietokantaa,
  et mallia.
</Callout>

### Chunkkausstrategiat

Datan pilkkominen on kriittinen vaihe — liian isot palat hukuttavat
relevantin tiedon, liian pienet menettävät kontekstin.

| Strategia | Kuvaus | Parhaiten sopii |
|---|---|---|
| **Kiinteä pituus** | N tokenia per chunk, limitys | Yksinkertainen teksti |
| **Lauseraja** | Pilkkoo virkkeiden mukaan | Luonnollinen teksti |
| **Kappaleraja** | Pilkkoo kappaleiden mukaan | Dokumentit, artikkelit |
| **Semanttinen** | Embedding-pohjainen rajaus | Monimutkainen sisältö |
| **Rekursiivinen** | Useita rajoja, valitsee parhaan | Yleiskäyttöinen |

<AnalogyBox>
  **Analogia:** RAG on kuin kirjastonhoitaja, joka hakee sinulle oikeat
  kirjat ennen kuin kirjoitat esseen. Sen sijaan, että yrität muistaa
  kaiken ulkoa (hienosäätö), menet kirjastoon (vektoritietokanta),
  pyydät apua (haku), ja kirjastonhoitaja hakee oikeat kirjat (tulokset).
  Sitten kirjoitat esseen kirjojen pohjalta (generointi).

  **Hyvä kirjastonhoitaja** tietää tarkalleen mitkä kirjat ovat relevantteja
  ja osaa jättää epäolennaiset pois. **Huono kirjastonhoitaja** tuo
  satunnaisia kirjoja tai liian monta kirjaa, jolloin hukut tietoon.
</AnalogyBox>

### Kehittyneet RAG-mallit

Perus-RAG on vasta alkua. Käytännössä tarvitaan usein monimutkaisempia
malleja:

#### Agentic RAG

Agentti päättää *miten* haku suoritetaan:

- **Reititys:** eri kyselyt → eri tietokannat (teksti, koodi, taulukot)
- **Kyselyn hajotus:** monimutkainen kysymys pilkotaan osiin
- **Iteraatio:** haku → tarkenna → haku uudelleen → generointi

```typescript
async function agenticRAG(query: string) {
  // 1. Analysoi kyselyn tyyppi
  const type = await classifyQuery(query);

  // 2. Reititä oikeaan tietokantaan
  if (type === "factual") {
    return searchVectorDB(query);
  } else if (type === "tabular") {
    return searchTabularDB(query);
  } else {
    return searchHybrid(query);
  }

  // 3. Tarkenna tarvittaessa
  // 4. Generoi vastaus
}
```

#### Multimodaalinen RAG

Tekstin lisäksi haetaan kuvia, taulukoita ja ääntä:

- Teksti upotetaan tekstienkooderilla (esim. text-embedding-3-small)
- Kuvat upotetaan kuvienkooderilla (esim. CLIP)
- Yhteinen upotusavaruus mahdollistaa "näytä kuva, joka muistuttaa tätä tekstiä"

#### Graph RAG

Tietograafit + vektorihaku:

- Entiteetit ja niiden väliset suhteet tallennetaan graafiin
- Vektorihaku löytää relevantit entiteetit
- Graafihaku löytää niiden väliset yhteydet
- Lopputulos: konteksti, joka ymmärtää *suhteet* eikä vain *samankaltaisuutta*

### RAG:n vikatilat — ja miten ne korjataan

| Vikatila | Oire | Syy | Ratkaisu |
|---|---|---|---|
| **Chunk liian pieni** | Konteksti puuttuu, vastaus on pinnallinen | Liian aggressiivinen chunkkaus | Kasvata chunk-kokoa, lisää kontekstin limitystä |
| **Chunk liian suuri** | Relevantti tieto hukkuu turhaan tekstiin | Liian isot palat | Pienennä chunkkeja, käytä semanttista chunkkausta |
| **Haku epäonnistuu** | Väärät dokumentit haetaan | Upotusmalli ei ymmärrä domainia | Vaihda upotusmalli (esim. domain-spesifi), lisää hakuun avainsanoja |
| **Malli ei seuraa kontekstia** | LLM vastaa omasta päästään, ei dokumenttien perusteella | Heikko prompt tai liian vähäinen konteksti | Tarkenna promptia: "Vastaa VAIN annetun kontekstin perusteella" |
| **Liian monta dokumenttia** | Token-budjetti ylittyy, vastaus sekava | Kontekstiin mahtuu rajoitettu määrä | Paranna haun tarkkuutta, käytä rankkausta (reranking) |

### Milloin EI käyttää RAG:ia

RAG on mahtava työkalu, mutta ei aina oikea valinta:

| Käytä RAG:ia... | Käytä hienosäätöä... | Käytä API-käärettä... |
|---|---|---|
| Data muuttuu usein (päivittäin/viikoittain) | Data on vakaa (kuukausia sama) | Et tarvitse omaa dataa ollenkaan |
| Tarvitset tarkkoja viittauksia lähteisiin | Tarvitset tietyn ulosoton muodon | Teet nopeaa prototyyppiä |
| Domain on erikoistunut (laki, lääketiede) | Haluat mallin oppivan tyylin | Kustannukset eivät ole kriittisiä |
| Hallusinaatioiden hinta on korkea | Haluat nopeamman latenssin | Et halua ylläpitää vektoritietokantaa |

### RAG:n evaluointi: miten mittaat laatua?

| Metriikka | Kuvaus | Tavoite |
|---|---|---|
| **precision@k** | Kuinka moni haetuista dokumenteista on relevantteja? | > 0.8 |
| **recall@k** | Kuinka suuri osa relevanteista dokumenteista löytyi? | > 0.7 |
| **NDCG** | Ovatko tärkeimmät dokumentit kärjessä? | > 0.85 |
| **Answer relevanssi** | Vastaako generoitu vastaus kysymykseen? | Manuaalinen / LLM-eval |
| **Faithfulness** | Onko vastaus uskollinen kontekstille (ei hallusinoi)? | LLM-eval |

### Suorituskykyvertailu

| Hakustrategia | Latenssi | Kustannus | Milloin käyttää |
|---|---|---|---|
| **Täysi RAG-haku** | 2-4s | Korkea | Ensimmäinen kysely, uniikki hakusana |
| **Semanttinen välimuisti** | 0.1-0.3s | Matala | Samankaltainen kysely toistuu (30-50% osuma) |
| **Tarkka välimuisti** | 0.05s | Minimissa | Täsmälleen sama kysely |
| **Streaming** | 0.1s eka token | Sama | Koettu nopeus, käyttäjä näkee vastauksen heti |

<WhyItMatters>
  **Miksi tällä on väliä?** RAG on yleisin tapa tuoda oma data osaksi
  AI-sovellusta. Se on halvempi kuin hienosäätö, helpompi päivittää kuin
  agentit, ja se vähentää hallusinaatioita merkittävästi. Oikea RAG-arkkitehtuuri
  on kuitenkin paljon monimutkaisempi kuin "laita dokumentti embedding-palveluun".
  Chunkkaus, haku ja kontekstin muodostus vaativat huolellista suunnittelua.
</WhyItMatters>

### Caching- ja streaming-strategiat

RAG-järjestelmän suorituskykyä voi parantaa merkittävästi:

- **Semanttinen cache:** samankaltaiset kyselyt palauttavat välimuistista
  (GPTCache, Redis + embeddings)
- **Vastauksen streaming:** tokenit välitetään käyttäjälle yksi kerrallaan
  koetun nopeuden parantamiseksi
- **Prefetching:** hae seuraava todennäköisin konteksti etukäteen

```typescript
// Semanttinen cache yksinkertaistettuna
async function getCachedOrFetch(query: string) {
  const queryEmbedding = await embed(query);
  const cached = await redis.similaritySearch(queryEmbedding, 0.95);

  if (cached) {
    return cached.answer; // Osuma! Palauta välimuistista
  }

  const answer = await fullRAGPipeline(query);
  await redis.store(queryEmbedding, answer);
  return answer;
}
```
