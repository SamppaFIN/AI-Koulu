## 4.1 Luotettavuus ja virheensieto

AI-järjestelmät ovat perusluonteeltaan epädeterministisiä. Tämä tarkoittaa,
että perinteiset virheenkäsittelymallit eivät toimi.

### Epädeterminististen komponenttien käsittely

```typescript
// Perinteinen: oleta että toimii
const result = api.call(data);
assert(result.isValid);

// AI-maailma: oleta että voi epäonnistua
const result = await ai.call(data);

// Tarkista ensin, luota sitten
if (!isValidResult(result)) {
  result = await fallbackStrategy(data);
}
```

### Fallback-strategiat

Kun LLM epäonnistuu, sinulla pitää olla suunnitelma:

| Taso | Strategia | Latenssi |
|---|---|---|
| **1** | Yritä uudelleen (retry) | +1× |
| **2** | Pienempi malli (gpt-4o → gpt-4o-mini) | −50% |
| **3** | Välimuistista (aiempi samankaltainen vastaus) | −90% |
| **4** | Sääntöpohjainen (template + parametrit) | −99% |
| **5** | Virheilmoitus käyttäjälle | — |

<Callout type="warning">
  **Tärkeää:** Fallback-strategia pitää testata *ennen* tuotantoonmenoa.
  Jos LLM epäonnistuu, tiedätkö mitä tapahtuu? Moni järjestelmä kaatuu
  hiljaa siksi, että fallbackia ei ole koskaan testattu.
</Callout>

### Uudelleenyrityslogiikka ja piirikatkaisijat

```typescript
// Piirikatkaisija (Circuit Breaker)
class CircuitBreaker {
  private failures = 0;
  private threshold = 5;
  private state: "closed" | "open" | "half-open" = "closed";

  async call(fn: () => Promise<any>) {
    if (this.state === "open") {
      throw new Error("Circuit breaker open");
    }
    try {
      const result = await fn();
      this.failures = 0;
      return result;
    } catch (e) {
      this.failures++;
      if (this.failures >= this.threshold) {
        this.state = "open";
        setTimeout(() => { this.state = "half-open"; }, 30000);
      }
      throw e;
    }
  }
}
```

### "Defensive design" probabilistisessa maailmassa

1. **Älä luota mihinkään** — tarkista jokainen vastaus
2. **Rajoita vahinkoja** — minimoi mitä malli voi tehdä
3. **Oleta pahinta** — malli hallusinoi, toimi sen mukaisesti

#### Robustnesstestaus

Miten mittaat järjestelmän kestävyyttä?

| Testi | Mitä testataan | Tavoite |
|---|---|---|
| **Graceful degradation** | Sammuta LLM API — toimiiko järjestelmä silti? | Fallback toimii < 2s |
| **Load test** | 100× normaali liikenne — kuinka monta virhettä? | < 5% virheitä |
| **Chaos testing** | Satunnaisia komponenttivikoja — selviääkö? | 100% toipuminen |
| **Adversarial test** | Lähetä jailbreak-yrityksiä — estääkö? | 0% onnistumista |

### Bias ja oikeudenmukaisuus

AI-mallit voivat toistaa harjoitusdatansa vinoumia. Arkkitehtuurin pitää havaita ne:

```python
# Yksinkertainen bias-testi: antaako malli eri vastauksia eri nimille?
names = ["Matti", "Fatima", "Wei", "Maria"]
for name in names:
    response = llm(f"Mitä uraa suosittelisit {name}:lle?")
    # Tarkista: onko vastauksissa systemaattisia eroja?
    # Esim. Matti → insinööri, Fatima → hoitaja → BIAS!
```

**Ratkaisu:**
- Lisää bias-esimerkkejä hienosäätödataan
- Käytä LLM-evalia: "Onko tämä vastaus puolueeton?"
- Kuukausittainen auditointi monimuotoisilla testitapauksilla

## 4.2 Skaalautuvuus

### Tilattomat vs. tilalliset komponentit

| Komponentti | Tila | Skaalautuvuus |
|---|---|---|
| **LLM-palvelin** | Tilaton (ei tallenna mitään) | Helppo, vaakasuora |
| **Vektoritietokanta** | Tilallinen (indeksi) | Vaikea, vaatii jakamista |
| **Keskusteluhistoria** | Tilallinen (säilytettävä) | Puskuroitu, Redis |
| **Agenttiprosessi** | Tilallinen (muisti) | Vaikea, istuntosidonnainen |

### Viestijonot ja asynkroninen käsittely

```typescript
// Event-driven AI: jono erottaa kutsun ja käsittelyn
interface AIRequest {
```

## 4.3 Turvallisuus ja vastuullinen AI

### Arkkitehtuuritason suojamekanismit

```typescript
// Syötevalidointi
function validateInput(input: string) {
  // 1. Estä jailbreak-yritykset
  if (containsJailbreak(input)) return reject();
  // 2. Poista PII
  input = removePII(input);
  // 3. Rajoita pituus
  if (input.length > 4000) return reject();
  return input;
}
```

### Sisäänrakennettu auditointi ja selitettävyys

```typescript
interface AuditLog {
```

### Pääsynhallinta

- **Kuka saa kutsua mitäkin mallia?** — kalliimmat mallit vaativat hyväksynnän
- **Kuka voi nähdä mitä dataa?** — sensitiivinen data rajatuille käyttäjille

### Tietovuotojen esto

- Älä koskaan lähetä sensitiivistä dataa kolmannen osapuolen API:in
- Anonymisoi ennen LLM-kutsua
- Tarkista vastauksista PII

## 4.4 Havainnoitavuus (Observability)

### Hajautettu jäljitys AI-putkessa

```typescript
// OpenTelemetry: jäljitä AI-kutsu
span.setAttribute("ai.prompt", prompt.substring(0, 100));
span.setAttribute("ai.model", "gpt-4o");
span.setAttribute("ai.tokens", completion.usage.total_tokens);
```

### Token-tason monitorointi

```
Graafi 1: Token-kulutus / käyttäjä / päivä
Graafi 2: Kustannus / käyttäjä / päivä
Graafi 3: Top-10 kalleimmat kyselyt
```

### Kustannusperäinen havainnointi

```sql
-- Esimerkki: kalleimmat käyttäjät
SELECT
  user_id,
  SUM(tokens) as total_tokens,
  SUM(cost) as total_cost,
  COUNT(*) as request_count
FROM ai_logs
WHERE date = CURRENT_DATE
GROUP BY user_id
ORDER BY total_cost DESC
LIMIT 10;
```

### Laatumetriikat

| Metriikka | Kuvaus | Miten mitataan |
|---|---|---|
| **Relevanssi** | Onko vastaus aiheeseen? | Käyttäjäpalaute, nosta/älä |
| **Factuality** | Onko vastaus totta? | Eval-joukko, manuaalinen tarkistus |
| **Turvallisuus** | Onko vastaus turvallinen? | Moderointi-API |
| **Kohteliaisuus** | Onko vastaus asiallinen? | Kielenkäytön analyysi |
