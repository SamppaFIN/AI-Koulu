## 5.1 Yritystason RAG-järjestelmä

Täydellinen arkkitehtuuri yrityksen dokumenttien kyselyjärjestelmälle.

### Arkkitehtuurikaavio

<Diagram type="enterprise-rag">
  {/* Datalähteet → ETL → Vektoritietokanta → API → UI */}
</Diagram>

### Komponentit

| Kerros | Teknologia | Tehtävä |
|---|---|---|
| **Datalähteet** | SharePoint, Confluence, S3 | Lähdedokumentit |
| **ETL-putki** | Airflow + Python | Hae, chunkkaa, upota |
| **Vektoritietokanta** | Pinecone / Weaviate | Indeksoi ja hae |
| **API** | FastAPI / Express | Reititä kyselyt |
| **UI** | React / Streamlit | Käyttöliittymä |
| **Monitorointi** | LangSmith / MLflow | Seuraa laatua |

### Skaalautuvuus

```yaml
# Kubernetes-konfiguraatio
apiVersion: apps/v1
kind: Deployment
metadata:
  name: rag-api
spec:
  replicas: 3
  resources:
    requests:
      memory: "1Gi"
    limits:
      memory: "2Gi"
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: rag-api-hpa
spec:
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

### Turvallisuus

- **Dokumenttitason pääsynhallinta:** käyttäjä näkee vain dokumentit, joihin hänellä on oikeus
- **PII-poisto:** automatisoi henkilötietojen poisto ennen indeksointia
- **Audit-loki:** kaikki kyselyt ja vastaukset lokitetaan

<Callout type="info">
  **Kriittinen oivallus:** Yritys-RAGissa suurin haaste ei ole tekniikka
  vaan *pääsynhallinta*. Jos et ole tarkkana, käyttäjä voi kysellä
  dokumenteista, joihin hänellä ei ole pääsyä, RAG:n kautta.
</Callout>

## 5.2 Asiakaspalvelun AI-agentti

### Putki

```
Puheentunnistus → LLM → Tietokantahaut → Puhesynteesi
```

### Teknologiavalinnat

| Vaihe | Teknologia | Vaihtoehtoja |
|---|---|---|
| **Puheentunnistus (STT)** | Whisper / Azure Speech | Speechmatics, Deepgram |
| **Päätely** | GPT-4o / Claude | Mikä tahansa LLM |
| **Tietokantahaku** | pgvector / Qdrant | Asiakkasdata, tilaukset |
| **Puhesynteesi (TTS)** | ElevenLabs / Azure Speech | OpenAI TTS, WellSaid |
| **Puhelinintegraatio** | Twilio / Vonage | Voximplant |

### Ihmiskontrolli ja eskalaatio

```typescript
// Eskalaatio ihmiselle
async function handleConversation(input: string, context: Context) {
  const intent = await classifyIntent(input);

  if (intent.confidence < 0.7) {
    // En ole varma → kysy tarkennusta
    return askClarification();
  }

  if (intent.type === "complaint" || intent.sentiment === "angry") {
    // Valitus tai vihainen asiakas → heti ihmiselle
    return escalateToHuman(intent);
  }

  const response = await getAIResponse(input, context);
  return response;
}
```

### "Kysy dokumentista" -tyylinen käyttöliittymä

```typescript
async function askDocument(question: string, documentId: string) {
  // 1. Hae relevantit tekstiosuudet
  const textChunks = await vectorDB.search(question, { documentId });

  // 2. Hae relevantit visuaaliset elementit
  const visuals = await vectorDB.searchVisual(question, { documentId });

  // 3. Yhdistä konteksti
  const context = buildMultimodalContext(textChunks, visuals);

  tarvitset ja miten ne voivat liittyä toisiinsa. Kopioi ideat, mukauta
  tarpeisiisi.

## 5.5 Tapahtumapohjainen AI-järjestelmä

Kun AI-järjestelmä altistuu suurelle määrälle asynkronisia tapahtumia,
tarvitaan tapahtumapohjainen arkkitehtuuri.

<Diagram type="event-driven-ai">
  {/* Events → Queue → Processor → AI → Storage → Notification → Client */}
</Diagram>

| Vaihe | Teknologia | Vaihtoehtoja |
|---|---|---|
| **Tapahtumien tuottaminen** | Webhook, API, IoT | Kafka Producer |
| **Viestijono** | Kafka, RabbitMQ | AWS SQS, Redis Streams |
| **Prosessointi** | Worker-pool (Python, Node.js) | Serverless (Lambda) |
| **AI-käsittely** | LLM API / lokaali malli | GPT-5, Llama 4 |
| **Tulosten tallennus** | PostgreSQL, S3 | Time-series DB |
| **Käyttäjänotifikaatio** | WebSocket, SSE, sähköposti | Push-notifikaatio |

```typescript
// Tapahtumapohjainen AI-putki
interface AIEvent {
  id: string;
  type: "document_uploaded" | "analysis_requested" | "alert_generated";
  payload: unknown;
  timestamp: Date;
}

// 1. Vastaanota tapahtuma
kafka.on('document_uploaded', async (event: AIEvent) => {
  // 2. Laita jonoon (hajautettu käsittely)
  await queue.enqueue(event);

  // 3. Worker käsittelee
  const result = await ai.process(event.payload);

  // 4. Tallenna tulos
  await db.save(result);

  // 5. Notifioi käyttäjä
  ws.send({ userId: event.userId, result });
});
```

<Callout type="tip">
  **Milloin valita tämä:** Kun käyttäjät odottavat vastauksia taustalla
  (ei reaaliajassa), tapahtumia tulee paljon ja ne vaihtelevat kooltaan.
  Esimerkki: 10 000 dokumentin analysointi yön aikana.
</Callout>

## 5.6 AI-pohjainen suosittelujärjestelmä

Yhdistä klassinen suosittelualgoritmi ja LLM-pohjainen personalisointi.

<Diagram type="recommendation-system">
  {/* Käyttäjädatan analysointi → CF-malli → LLM personalisointi → Suositukset → UI */}
</Diagram>

| Komponentti | Tarkoitus | Esimerkkiteknologia |
|---|---|---|
| **Collaborative filtering** | Löydä samankaltaisia käyttäjiä/tuotteita | Surprise, implicit, LightFM |
| **Sisällön ymmärtäminen** | Analysoi tuotekuvaukset, kuvat | CLIP, text-embedding |
| **LLM-pohjainen rankkaus** | Järjestä suositukset personoidusti | GPT-5, Claude 4 |
| **Selitykset** | Generoi "Miksi suosittelemme tätä?" | LLM |

```python
def recommend(user_id):
    # 1. Hae kandidatit CF-mallilla (nopea, ei AI:ta)
    candidates = cf_model.get_candidates(user_id, n=100)

    # 2. LLM arvioi personoidusti 100 → 10
    prompt = f"User preferences: {user_profile}\nCandidates: {candidates}"
    ranked = llm.rank(prompt, candidates, top_k=10)

    # 3. Generoi selitykset
    explanations = llm.explain(f"Why {ranked[0]} for user {user_id}?")

    return {"recommendations": ranked, "explanations": explanations}
